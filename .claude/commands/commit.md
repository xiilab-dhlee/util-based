# 커밋 명령어

Conventional Commits 규칙을 따르는 한글 커밋 메시지를 생성합니다.

## 실행 절차

1. `git status`와 `git diff`로 변경사항 분석
2. 변경사항을 기능/목적별로 그룹화
3. 각 그룹에 적합한 커밋 타입 선정
4. 한글로 커밋 메시지 작성 및 커밋 실행

## 커밋 타입 (Conventional Commits)

| 타입 | 설명 | 예시 |
|------|------|------|
| `feat` | 새로운 기능 추가 | feat(auth): 로그인 기능 구현 |
| `fix` | 버그 수정 | fix(api): 토큰 만료 오류 수정 |
| `docs` | 문서 변경 | docs: README 업데이트 |
| `style` | 코드 포맷팅, 세미콜론 누락 등 | style: 코드 포맷팅 적용 |
| `refactor` | 리팩토링 (기능 변경 없음) | refactor(utils): 헬퍼 함수 분리 |
| `perf` | 성능 개선 | perf(query): 쿼리 최적화 |
| `test` | 테스트 추가/수정 | test(auth): 로그인 테스트 추가 |
| `build` | 빌드 시스템, 외부 종속성 변경 | build: vite 설정 수정 |
| `ci` | CI 설정 변경 | ci: GitHub Actions 워크플로우 추가 |
| `chore` | 기타 변경사항 | chore: 불필요한 파일 삭제 |

## 커밋 메시지 형식

```
<타입>[(범위)]: <설명>

[본문 - 선택사항]

[꼬리말 - 선택사항]
```

## 작성 규칙

1. **설명**: 명령형 어조로 작성 (예: "추가", "수정", "삭제")
2. **범위**: 변경된 모듈/컴포넌트명 (선택사항)
3. **본문**: 변경 이유나 상세 내용 (선택사항)
4. **꼬리말**: BREAKING CHANGE, 이슈 번호 등 (선택사항)

## 예시

```
feat(account): 계정 삭제 모달 구현

- 삭제 확인 다이얼로그 추가
- 삭제 API 연동
```

```
fix(group): 그룹 목록 조회 시 무한 로딩 수정

useEffect 의존성 배열 누락으로 인한 문제 해결

Closes #123
```

```
refactor!: API 응답 구조 변경

BREAKING CHANGE: 모든 API 응답이 { data, meta } 형식으로 변경됨
```

## 실행

변경사항을 분석하고 기능별로 적절한 커밋을 생성하세요.
